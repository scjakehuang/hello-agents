## 一、数据类型与应用场景

### 1.1 五种基本类型

| 类型 | 底层编码 | 场景 |
|------|----------|------|
| String | int / embstr / raw | 缓存、计数器、分布式锁 |
| List | quicklist(ziplist+linkedlist) | 消息队列、最新列表 |
| Hash | ziplist / hashtable | 对象存储、购物车 |
| Set | intset / hashtable | 去重、交集并集、标签 |
| ZSet | ziplist / skiplist+hashtable | 排行榜、延迟队列 |

### 1.2 三种扩展类型

| 类型 | 说明 | 场景 |
|------|------|------|
| HyperLogLog | 基数估算，12KB 可估算 2^64 个元素 | UV 统计 |
| Bitmap | 位操作，1bit 存储一个布尔值 | 签到、在线状态 |
| Geo | 地理位置计算（ZSet 底层） | 附近的人、距离计算 |

## 二、持久化

### 2.1 RDB vs AOF

```
┌──────────────────────────────────────────────────────────┐
│                  RDB vs AOF                               │
├──────────────────────┬───────────────────────────────────┤
│        RDB           │              AOF                   │
├──────────────────────┼───────────────────────────────────┤
│ 全量快照             │ 增量日志                           │
│ 二进制文件，体积小   │ 文本日志，体积大                   │
│ 恢复速度快           │ 恢复速度慢                         │
│ 可能丢数据（两次间隔）│ 最多丢1秒数据（everysec）         │
│ fork 子进程，不阻塞  │ 写入追加，rewrite 时可能阻塞       │
│ save/bgsave 触发    │ always/everysec/no                 │
└──────────────────────┴───────────────────────────────────┘
```

### 2.2 RDB 详解

```
┌──────────────────────────────────────────────────────────┐
│                   RDB 快照流程                            │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  触发方式：                                               │
│  ├── save      同步阻塞，生产禁用                        │
│  ├── bgsave    fork 子进程，不阻塞（推荐）                │
│  └── 自动触发  save 900 1 / save 300 10 / save 60 10000  │
│                                                           │
│  bgsave 流程：                                            │
│  1. 主进程 fork() 子进程（copy-on-write）                 │
│  2. 子进程遍历内存生成 RDB 文件                           │
│  3. 主进程继续处理命令                                    │
│  4. 子进程完成后替换旧 RDB 文件                           │
│                                                           │
│  COW（Copy-On-Write）：                                   │
│  ├── fork 后子进程共享父进程内存页                        │
│  ├── 主进程修改某页时，OS 复制该页给主进程                 │
│  └── 子进程看到的还是 fork 时的数据                       │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 2.3 AOF 详解

```
┌──────────────────────────────────────────────────────────┐
│                   AOF 写入流程                            │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  写入策略：                                               │
│  ├── always    每条命令 fsync，最安全，最慢               │
│  ├── everysec  每秒 fsync，最多丢1秒数据（推荐）         │
│  └── no        OS 决定何时 fsync，最快，可能丢数据        │
│                                                           │
│  写入流程：                                               │
│  命令 → AOF Buffer → fsync → AOF File                   │
│                                                           │
│  AOF Rewrite（重写）：                                    │
│  ├── 触发：auto-aof-rewrite-percentage 100               │
│  │       auto-aof-rewrite-min-size 64mb                  │
│  ├── 原理：fork 子进程，遍历当前数据生成最简命令          │
│  ├── 期间新命令同时写入旧AOF和重写缓冲区                  │
│  └── 重写完成后：新AOF + 重写缓冲区 → 替换旧AOF          │
│                                                           │
│  AOF 混合持久化（4.0+）：                                 │
│  ├── rewrite 后前半段是 RDB 格式（快）                    │
│  └── 后半段是 AOF 增量（不丢数据）                        │
│  aof-use-rdb-preamble yes                                 │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

## 三、内存管理

### 3.1 过期策略

```
┌──────────────────────────────────────────────────────────┐
│                 过期 Key 删除策略                          │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  惰性删除（Lazy Free）：                                   │
│  ├── 访问 Key 时才检查是否过期                            │
│  ├── 优点：CPU 友好                                       │
│  └── 缺点：过期 Key 不访问则一直占内存                    │
│                                                           │
│  定期删除（Periodic）：                                    │
│  ├── 每 100ms 随机抽取部分设置了过期的 Key                │
│  ├── 检查并删除过期的 Key                                  │
│  ├── 如果过期比例 > 25%，继续抽查                         │
│  └── 限制执行时长，避免阻塞                               │
│                                                           │
│  Redis 实际：惰性删除 + 定期删除 组合使用                 │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 3.2 内存淘汰策略

| 策略 | 说明 |
|------|------|
| noeviction | 不淘汰，内存满拒绝写入（默认） |
| allkeys-lru | 所有Key中LRU淘汰 |
| allkeys-lfu | 所有Key中LFU淘汰（4.0+） |
| allkeys-random | 所有Key中随机淘汰 |
| volatile-lru | 设了过期的Key中LRU淘汰 |
| volatile-lfu | 设了过期的Key中LFU淘汰 |
| volatile-random | 设了过期的Key中随机淘汰 |
| volatile-ttl | 设了过期的Key中TTL小的优先淘汰 |

**面试题：如何选择淘汰策略？**

```
有明确冷热区分 → allkeys-lru
有明确访问频率 → allkeys-lfu
所有Key同等重要 → allkeys-random
只有部分Key可淘汰 → volatile-lru
数据不能丢 → noeviction + 扩容
```

### 3.3 缓存问题

```
┌──────────────────────────────────────────────────────────┐
│                  缓存三大问题                              │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  1. 缓存穿透（查询不存在的数据）                          │
│     ├── 缓存空值（设短过期时间）                          │
│     ├── 布隆过滤器（拦截不存在的Key）                     │
│     └── 参数校验（拦截非法请求）                          │
│                                                           │
│  2. 缓存击穿（热点Key过期瞬间大量请求打到DB）             │
│     ├── 互斥锁（setnx抢锁，只允许一个线程重建缓存）       │
│     ├── 逻辑过期（不设TTL，数据中存过期时间）             │
│     └── 热点Key永不过期（手动更新）                       │
│                                                           │
│  3. 缓存雪崩（大量Key同时过期）                           │
│     ├── 过期时间加随机值，打散过期时间                    │
│     ├── 多级缓存（本地缓存 + Redis）                     │
│     ├── 熔断降级（DB不可用时空窗降级）                    │
│     └── Redis 高可用（哨兵/集群）                        │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

## 四、高可用

### 4.1 主从复制

```
┌──────────────────────────────────────────────────────────┐
│                  主从复制流程                              │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  全量同步（首次连接）：                                    │
│  1. Slave 发送 psync ? -1（首次）                         │
│  2. Master 执行 bgsave 生成 RDB                           │
│  3. Master 发送 RDB 给 Slave                              │
│  4. Slave 清空数据，加载 RDB                              │
│  5. Master 发送缓冲区的增量命令给 Slave                   │
│                                                           │
│  增量同步（断线重连）：                                    │
│  1. Slave 发送 psync {runId} {offset}                     │
│  2. Master 检查 offset 是否在 repl_backlog 中             │
│     ├── 在 → 发送 offset 之后的增量数据                   │
│     └── 不在 → 退化为全量同步                              │
│                                                           │
│  repl_backlog_size 默认 1MB                               │
│  → 如果主从断开期间写入超过 1MB → 全量同步                │
│  → 生产建议调大到 256MB+                                  │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 4.2 哨兵模式

```
┌──────────────────────────────────────────────────────────┐
│                  Sentinel 故障转移                        │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  1. 主观下线（SDOWN）                                     │
│     单个 Sentinel 认为 Master 不可用                      │
│                                                           │
│  2. 客观下线（ODOWN）                                     │
│     超过 quorum 个 Sentinel 认为 Master 不可用            │
│                                                           │
│  3. Sentinel Leader 选举                                   │
│     通过 Raft 算法选出一个 Sentinel 执行故障转移           │
│                                                           │
│  4. 新 Master 选举                                        │
│     ├── 排除下线的 Slave                                  │
│     ├── 优先选 replica-priority 最小的                    │
│     ├── 优先选复制偏移量最大的（数据最新）                 │
│     └── 优先选 runId 最小的                               │
│                                                           │
│  5. 故障转移                                              │
│     ├── 对新 Master 执行 slaveof no one                   │
│     ├── 通知其他 Slave 复制新 Master                      │
│     └── 监控旧 Master，恢复后设为新 Master 的 Slave       │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

```

├──────────────────────────────────────────────────────────┤

│                                                           │

│  └── 最终一致性                                           │
│                                                           │
```

## 五、应用模式

### 5.1 分布式锁

```java
// 加锁（SETNX + 过期时间，原子操作）
String result = jedis.set(lockKey, requestId, "NX", "EX", 30);

// 释放锁（Lua 脚本保证原子性）
String script = "if redis.call('get', KEYS[1]) == ARGV[1] " +
    "then return redis.call('del', KEYS[1]) " +
    "else return 0 end";
jedis.eval(script, Collections.singletonList(lockKey),
    Collections.singletonList(requestId));

// Redisson 分布式锁（推荐）
RLock lock = redisson.getLock("order:123");
lock.lock(30, TimeUnit.SECONDS);
try {
    // 业务逻辑
} finally {
    lock.unlock();
}

// Redisson 看门狗机制：
// lock() 不设过期时间时，默认 30 秒
// 每 10 秒（1/3 leaseTime）自动续期
// 业务完成 unlock 或客户端宕机才停止续期
```

### 5.2 Redisson vs 手动实现

| 特性 | 手动 SETNX | Redisson |
|------|-----------|----------|
| 可重入 | 不支持 | 支持（Hash结构记录重入次数） |
| 自动续期 | 不支持 | 看门狗自动续期 |
| 公平锁 | 不支持 | RedissonFairLock |
| 读写锁 | 不支持 | RedissonReadWriteLock |
| 信号量 | 不支持 | RedissonSemaphore |
| RedLock | 不支持 | 多节点加锁 |

## 六、高频面试题

| 问题 | 核心答案 |
|------|----------|
| Redis 为什么快？ | 内存操作 + 单线程无锁 + IO多路复用 + 高效数据结构 |
| RDB 和 AOF 怎么选？ | 对数据安全要求高用AOF，可容忍几分钟丢失用RDB，推荐混合持久化 |
| 缓存穿透/击穿/雪崩？ | 穿透：布隆过滤器/空值缓存；击穿：互斥锁/逻辑过期；雪崩：随机TTL/多级缓存 |
| Redis Cluster 为什么是 16384 个 Slot？ | 16384 足够均匀分布，Gossip 包大小可控（2KB），集群节点上限 1000 |
| 如何保证缓存和数据库一致性？ | 先更新DB再删缓存 + 延迟双删 / 订阅Binlog异步删缓存 |
| 热Key问题怎么解决？ | 本地缓存 + 热Key打散（加随机后缀分布到多个Key） |
| BigKey 怎么处理？ | 拆分为小Key；用hash结构代替大string；unlink异步删除 |
