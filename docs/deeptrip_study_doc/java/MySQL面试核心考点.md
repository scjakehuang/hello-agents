## 一、存储引擎

### 1.1 InnoDB vs MyISAM

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务 | 支持 | 不支持 |
| 锁粒度 | 行锁 | 表锁 |
| 外键 | 支持 | 不支持 |
| MVCC | 支持 | 不支持 |
| 崩溃恢复 | 支持（redo log） | 不支持 |
| 聚簇索引 | 是 | 否 |
| 全文索引 | 5.6+ 支持 | 支持 |
| COUNT(*) | 需遍历 | 存储了行数 |

**面试题：为什么选 InnoDB？** → 事务 + 行锁 + MVCC + 崩溃恢复

### 1.2 InnoDB 架构

```
┌──────────────────────────────────────────────────────────────┐
│                    InnoDB 内存架构                             │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  ┌──────────────────────────────────────────────────┐        │
│  │                  Buffer Pool                      │        │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐         │        │
│  │  │Data Page │ │Index Page│ │Undo Page │         │        │
│  │  └──────────┘ └──────────┘ └──────────┘         │        │
│  │  ┌──────────┐ ┌──────────┐                      │        │
│  │  │Insert Buf│ │Adaptive  │                      │        │
│  │  │(非聚簇)   │ │Hash Index│                      │        │
│  │  └──────────┘ └──────────┘                      │        │
│  └──────────────────────────────────────────────────┘        │
│                                                               │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │  Redo Log    │  │  Undo Log    │  │  Log Buffer  │       │
│  │  Buffer      │  │  (回滚)       │  │  (写缓冲)    │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
│                                                               │
└──────────────────────────────────────────────────────────────┘

写入流程：
  1. 修改数据页 → Buffer Pool（内存）
  2. 记录修改 → Redo Log Buffer
  3. Redo Log Buffer → Redo Log File（WAL，先写日志再写数据）
  4. 异步刷脏页 → 数据文件
```

## 二、索引

### 2.1 索引数据结构

```
B+ Tree 为什么适合数据库索引？

┌──────────────────────────────────────────────────────────────┐
│  B+ Tree 特点：                                               │
│  1. 非叶子节点只存键值，不存数据 → 每页容纳更多键 → 树更矮   │
│  2. 所有数据在叶子节点 → 查询性能稳定 O(logN)                │
│  3. 叶子节点链表相连 → 范围查询高效                           │
│                                                               │
│  三层 B+ Tree 可存储：                                        │
│  每页约1000个键 × 1000 × 1000 = 10亿行                       │
│  三次IO即可定位任意一行                                       │
│                                                               │
│           [30 | 60]                                           │
│          /    |    \                                          │
│   [10|20]  [40|50]  [70|80|90]                               │
│   / | \    / | \    / | \  |                                 │
│  →10→20→  →30→40→50→  →60→70→80→90→                         │
│  (叶子节点双向链表，范围扫描高效)                               │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 聚簇索引 vs 非聚簇索引

| 特性 | 聚簇索引 | 非聚簇索引（二级索引） |
|------|----------|----------------------|
| 叶子节点存储 | 完整行数据 | 主键值 |
| 每表数量 | 1个 | 多个 |
| 查询方式 | 直接获取数据 | 回表查询（通过主键再查聚簇索引） |
| 插入顺序 | 按主键有序 | 不影响 |

### 2.3 索引优化

**覆盖索引**：

```sql
-- 查询只需要索引列，不需要回表
SELECT user_id, order_time FROM orders WHERE user_id = 100;
-- 索引 (user_id, order_time) 可覆盖查询

-- EXPLAIN 中 Extra = Using index 表示覆盖索引
```

**最左前缀原则**：

```sql
-- 联合索引 (a, b, c)
-- 有效查询：
WHERE a = 1                    ✓ 使用 a
WHERE a = 1 AND b = 2          ✓ 使用 a, b
WHERE a = 1 AND b = 2 AND c = 3  ✓ 使用 a, b, c
WHERE a = 1 AND c = 3          ✓ 使用 a（c 无法使用索引）

-- 无效查询：
WHERE b = 2                    ✗ 不走索引
WHERE b = 2 AND c = 3          ✗ 不走索引
WHERE c = 3                    ✗ 不走索引

-- 范围查询后的列无法使用索引：
WHERE a > 1 AND b = 2          ✓ 只使用 a（范围后中断）
```

**索引下推（ICP, Index Condition Pushdown）**：

```sql
-- 联合索引 (name, age)
SELECT * FROM users WHERE name LIKE '张%' AND age = 25;

-- 无 ICP：存储引擎通过 name 找到行 → 回表 → Server 层过滤 age
-- 有 ICP：存储引擎在索引中直接过滤 age → 减少回表次数
-- EXPLAIN 中 Extra = Using index condition
```

## 三、事务与锁

### 3.1 ACID 与实现

| 特性 | 实现 |
|------|------|
| 原子性 | Undo Log（回滚） |
| 一致性 | 原子性 + 隔离性 + 持久性 共同保证 |
| 隔离性 | MVCC + 锁 |
| 持久性 | Redo Log（WAL） |

### 3.2 MVCC

```
┌──────────────────────────────────────────────────────────┐
│                    MVCC 实现原理                          │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  核心组件：                                               │
│  1. 隐藏字段                                              │
│     ├── DB_TRX_ID     最近修改的事务ID                    │
│     ├── DB_ROLL_PTR   指向 undo log 的回滚指针            │
│     └── DB_ROW_ID     隐藏自增ID（无主键时）              │
│                                                           │
│  2. Undo Log 版本链                                       │
│     ┌──────────┐    ┌──────────┐    ┌──────────┐         │
│     │ trx_id=3 │◀───│ trx_id=2 │◀───│ trx_id=1 │         │
│     │ 当前版本  │    │ 历史版本  │    │ 历史版本  │         │
│     └──────────┘    └──────────┘    └──────────┘         │
│                                                           │
│  3. Read View（读视图）                                   │
│     ├── creator_trx_id   创建该 Read View 的事务ID        │
│     ├── m_ids            创建时所有活跃事务ID列表          │
│     ├── min_trx_id       活跃事务最小ID                   │
│     └── max_trx_id       下一个将分配的事务ID              │
│                                                           │
│  可见性判断：                                              │
│  trx_id < min_trx_id          → 可见（事务已提交）        │
│  trx_id >= max_trx_id         → 不可见（后开始的事务）    │
│  min_trx_id <= trx_id < max   → 不在 m_ids 中则可见      │
│                                                           │
│  RC 级别：每次 SELECT 创建新的 Read View                   │
│  RR 级别：第一次 SELECT 创建 Read View，后续复用           │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 3.3 锁机制

```
┌──────────────────────────────────────────────────────────┐
│                  InnoDB 锁类型                            │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  全局锁     FLUSH TABLES WITH READ LOCK                   │
│             → 全库只读，用于备份                           │
│                                                           │
│  表级锁     ├── 表锁：LOCK TABLES ... READ/WRITE          │
│             ├── 元数据锁（MDL）：防 DDL 与 DML 冲突        │
│             ├── 意向锁（IS/IX）：表级标记，快速判断        │
│             └── AUTO-INC 锁：自增ID分配                   │
│                                                           │
│  行级锁     ├── Record Lock：锁定索引记录                 │
│             ├── Gap Lock：锁定索引间隙（开区间）           │
│             └── Next-Key Lock：Record + Gap（左开右闭）   │
│                                                           │
│  加锁规则（丁奇总结，MySQL 8.0验证）：                     │
│  1. 加锁的基本单位是 Next-Key Lock                        │
│  2. 查找过程中访问到的对象才会加锁                         │
│  3. 等值查询唯一索引，Next-Key 退化为 Record Lock         │
│  4. 等值查询普通索引，向右遍历到最后一个不满足条件时，      │
│     Next-Key 退化为 Gap Lock                              │
│  5. 唯一索引上的范围查询会访问到不满足条件的第一个值为止    │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

## 四、SQL 优化

### 4.1 EXPLAIN 关键字段

| 字段 | 说明 | 关注点 |
|------|------|--------|
| type | 访问类型 | system > const > eq_ref > ref > range > index > ALL |
| key | 实际用到的索引 | 是否命中索引 |
| rows | 预估扫描行数 | 越小越好 |
| Extra | 额外信息 | Using index / Using filesort / Using temporary |

### 4.2 慢查询优化思路

```
1. 开启慢查询日志
   slow_query_log = ON
   long_query_time = 1

2. EXPLAIN 分析执行计划

3. 常见优化手段
   ├── 添加合适索引
   ├── 避免索引失效（函数计算、隐式转换、LIKE '%xx'、OR）
   ├── 覆盖索引减少回表
   ├── 分页优化（游标分页代替 OFFSET）
   ├── 大表 DDL 用 pt-online-schema-change
   └── 数据归档（冷热分离）
```

## 五、高频面试题

| 问题 | 核心答案 |
|------|----------|
| 为什么用 B+ Tree 不用 B Tree？ | B+ 非叶节点不存数据 → 更矮 → IO更少；叶子链表 → 范围查询快 |
| 什么是回表？ | 二级索引查到主键 → 再查聚簇索引获取完整行 |
| MVCC 怎么实现的？ | 隐藏字段 + Undo Log 版本链 + Read View 可见性判断 |
| RR 级别怎么解决幻读？ | Next-Key Lock（加间隙锁防止插入）；MVCC 快照读 |
| 索引什么时候失效？ | 函数/计算/隐式转换/LIKE左模糊/不满足最左前缀/OR |
| Redo Log 和 Undo Log 的区别？ | Redo 保证持久性（前滚）；Undo 保证原子性（回滚） |
