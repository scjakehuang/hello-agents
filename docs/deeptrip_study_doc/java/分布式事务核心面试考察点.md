## 一、为什么需要分布式事务

### 1.1 从本地事务到分布式事务

```
单体架构                          微服务架构
┌────────────────────┐           ┌──────────┐  ┌──────────┐  ┌──────────┐
│  Application       │           │ Order    │  │ Payment  │  │ Inventory│
│  ┌──────────────┐  │           │ Service  │  │ Service  │  │ Service  │
│  │   本地事务    │  │           └────┬─────┘  └────┬─────┘  └────┬─────┘
│  │  ACID 保证   │  │                │              │              │
│  └──────────────┘  │                ▼              ▼              ▼
│  ┌───────┐         │           ┌────────┐    ┌────────┐    ┌────────┐
│  │  DB   │         │           │Order DB│    │Pay  DB │    │Inv  DB │
│  └───────┘         │           └────────┘    └────────┘    └────────┘
└────────────────────┘
  一个事务，一个DB               多个事务，多个DB
  ACID 天然保证                  需要分布式事务保证一致性
```

### 1.2 典型场景

| 场景 | 涉及服务 | 一致性要求 |
|------|----------|------------|
| 电商下单 | 订单 + 库存 + 支付 | 强一致（库存不能超卖） |
| 转账 | 转出账户 + 转入账户 | 强一致（金额不能丢） |
| 订单超时取消 | 订单 + 库存 + 优惠券 | 最终一致（允许短暂不一致） |
| 跨行清算 | 行内账户 + 人行清算 | 最终一致（T+1 对账兜底） |

### 1.3 CAP 与 BASE

```
CAP 定理：分布式系统最多同时满足两个

        ┌─────────────┐
        │  Consistency │  一致性
        └──────┬──────┘
               │
    ┌──────────┼──────────┐
    │          │          │
    ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌──────────┐
│ CA     │ │ CP     │ │ AP       │
│ 单机RDB│ │ Zookeeper│ │ Cassandra│
│ 不分区 │ │ 强一致  │ │ 高可用   │
└────────┘ └────────┘ └──────────┘
    │          │          │
    └──────────┼──────────┘
               │
        ┌──────┴──────┐
        │ Availability│  可用性
        └─────────────┘
               │
        ┌──────┴──────┐
        │Partition to- │  分区容错
        │lerance       │
        └─────────────┘
```

**BASE 理论**（分布式事务的理论基础）：

| 原则 | 含义 |
|------|------|
| **B**asically Available | 基本可用，允许响应时间增加或功能降级 |
| **S**oft state | 软状态，允许中间状态存在 |
| **E**ventually consistent | 最终一致性，一段时间后达到一致 |

## 三、2PC / XA 协议

### 3.1 原理

```
                    协调者 (Coordinator / TM)
                         │
        ┌────────────────┼────────────────┐
        │                │                │
        ▼                ▼                ▼
   ┌─────────┐     ┌─────────┐     ┌─────────┐
   │ 参与者 A │     │ 参与者 B │     │ 参与者 C │
   │ (RM)    │     │ (RM)    │     │ (RM)    │
   └─────────┘     └─────────┘     └─────────┘

阶段一：Prepare（准备）
  协调者 ──▶ 参与者: "请准备好提交"
  参与者 ──▶ 协调者: "已就绪 / 失败"

阶段二：Commit / Rollback（提交/回滚）
  全部就绪 ──▶ 协调者 ──▶ 参与者: "提交"
  任一失败 ──▶ 协调者 ──▶ 参与者: "回滚"
```

### 3.3 面试高频问题

**Q1: 2PC 有什么缺陷？**

| 问题 | 说明 |
|------|------|
| **同步阻塞** | Prepare 阶段持有锁，直到 Commit/Rollback 释放，性能差 |
| **单点故障** | 协调者宕机，参与者永远阻塞 |
| **数据不一致** | 阶段二网络分区，部分参与者提交、部分未提交 |
| **太过保守** | 任一参与者超时未响应，整个事务回滚 |

**Q2: 3PC 对 2PC 做了什么改进？为什么还是不用？**

```
3PC 三阶段：
  1. CanCommit   ──▶ 询问，不持有锁（减少阻塞）
  2. PreCommit   ──▶ 准备，持有锁
  3. DoCommit    ──▶ 提交

改进：增加超时机制，参与者超时后自动提交（降低阻塞）

仍未解决的问题：
  - 网络分区下，超时自动提交仍可能导致数据不一致
  - 多一轮交互，延迟更高
  - 工程实现复杂，收益有限
```

## 四、TCC 模式

### 4.1 原理

```
TCC = Try - Confirm - Cancel

┌──────────────────────────────────────────────────────┐
│                    TCC 事务流程                        │
├──────────────────────────────────────────────────────┤
│                                                       │
│   Try 阶段（资源预留）                                 │
│   ├── 冻结库存（不真正扣减）                           │
│   ├── 冻结账户余额                                     │
│   └── 创建订单（状态: INIT）                           │
│                                                       │
│   全部 Try 成功？                                      │
│   ├── YES ──▶ Confirm 阶段（确认提交）                │
│   │          ├── 扣减冻结库存                          │
│   │          ├── 扣减冻结余额                          │
│   │          └── 订单状态 → SUCCESS                    │
│   │                                                    │
│   └── NO ───▶ Cancel 阶段（取消回滚）                 │
│              ├── 释放冻结库存                          │
│              ├── 释放冻结余额                          │
│              └── 订单状态 → CANCELLED                  │
│                                                       │
└──────────────────────────────────────────────────────┘
```

### 4.2 代码实现

```java
public interface InventoryTccService {

    /**
     * Try: 冻结库存
     */
    @TwoPhaseBusinessAction(
        name = "deductInventory",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    boolean tryDeduct(
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        @BusinessActionContextParameter(paramName = "count") Integer count
    );

    /**
     * Confirm: 确认扣减
     */
    boolean confirm(BusinessActionContext context);

    /**
     * Cancel: 释放冻结
     */
    boolean cancel(BusinessActionContext context);
}

@Service
public class InventoryTccServiceImpl implements InventoryTccService {

    @Autowired
    private InventoryMapper inventoryMapper;
    @Autowired
    private FreezeMapper freezeMapper;

    @Override
    public boolean tryDeduct(Long productId, Integer count) {
        // 1. 检查可用库存
        Inventory inv = inventoryMapper.selectByProductId(productId);
        if (inv.getAvailable() < count) {
            throw new RuntimeException("库存不足");
        }

        // 2. 冻结库存（资源预留）
        inventoryMapper.freeze(productId, count);

        // 3. 记录冻结明细（用于 Confirm/Cancel）
        FreezeRecord record = new FreezeRecord()
            .setProductId(productId)
            .setCount(count)
            .setStatus("FROZEN");
        freezeMapper.insert(record);

        return true;
    }

    @Override
    public boolean confirm(BusinessActionContext context) {
        Long productId = (Long) context.getActionContext("productId");
        Integer count = (Integer) context.getActionContext("count");

        // 幂等检查
        FreezeRecord record = freezeMapper.selectByContext(context.getXid());
        if (record == null || "CONFIRMED".equals(record.getStatus())) {
            return true;
        }

        // 真正扣减库存
        inventoryMapper.deduct(productId, count);
        // 更新冻结记录
        freezeMapper.updateStatus(record.getId(), "CONFIRMED");
        return true;
    }

    @Override
    public boolean cancel(BusinessActionContext context) {
        Long productId = (Long) context.getActionContext("productId");
        Integer count = (Integer) context.getActionContext("count");

        // 幂等检查
        FreezeRecord record = freezeMapper.selectByContext(context.getXid());
        if (record == null || "CANCELLED".equals(record.getStatus())) {
            return true;
        }

        // 释放冻结库存
        inventoryMapper.unfreeze(productId, count);
        // 更新冻结记录
        freezeMapper.updateStatus(record.getId(), "CANCELLED");
        return true;
    }
}
```

### 4.3 TCC 三大难题

```
┌──────────────────────────────────────────────────────────────┐
│                    TCC 三大难题与解决方案                      │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. 空回滚                                                    │
│     问题: Try 未执行，Cancel 却被调用                          │
│     原因: Try 请求网络丢失，事务管理器触发回滚                  │
│     解决: Cancel 检查 Try 是否执行过，未执行则直接返回成功      │
│                                                               │
│  2. 幂等性                                                    │
│     问题: Confirm/Cancel 被重复调用                            │
│     原因: 网络超时重试                                        │
│     解决: 通过全局事务ID + 状态做幂等判断                      │
│                                                               │
│  3. 悬挂                                                      │
│     问题: Cancel 比 Try 先执行，Try 之后又执行了               │
│     原因: Try 请求延迟到达                                    │
│     解决: Try 执行前检查是否已经 Cancel 过                     │
│                                                               │
│  ┌─────────────────────────────────────────┐                 │
│  │  防悬挂表设计                            │                 │
│  │  ┌──────────┬────────────┬──────────┐   │                 │
│  │  │ xid      │ branch_id  │ status   │   │                 │
│  │  ├──────────┼────────────┼──────────┤   │                 │
│  │  │ tx-001   │ branch-01  │ trying   │   │                 │
│  │  │ tx-001   │ branch-01  │ cancelled│   │                 │
│  │  └──────────┴────────────┴──────────┘   │                 │
│  │  Try 前插入 trying，Cancel 时更新为       │                 │
│  │  cancelled；Try 发现已 cancelled 则跳过   │                 │
│  └─────────────────────────────────────────┘                 │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## 五、Saga 模式

### 5.1 原理

```
Saga = 一串本地事务 + 补偿事务

正向执行：
  T1 ──▶ T2 ──▶ T3 ──▶ T4  (全部成功)

补偿回滚（逆向执行）：
  T1 ──▶ T2 ──▶ T3 ✗
                  │
                  ▼
         C3 ──▶ C2 ──▶ C1  (按反序补偿)

其中 Ti 是第 i 步正向事务，Ci 是 Ti 的补偿事务
```

### 5.2 两种编排方式

** Choreography（事件驱动编排）**：

```
┌─────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Order   │     │ Payment  │     │Inventory │     │ Shipping │
│ Service  │     │ Service  │     │ Service  │     │ Service  │
└────┬─────┘     └────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │                │
     │──OrderCreated──▶               │                │
     │                │──PaymentDone──▶                │
     │                │                │──Reserved────▶│
     │                │                │                │
     │                │                │  ✗ ShippingFailed
     │                │                │◀──CancelRes───│
     │                │◀─RefundPayment─│                │
     │◀─CancelOrder───│                │                │

优点：松耦合，无单点
```

## 六、Seata AT 模式

### 6.1 原理

```
Seata AT = 无侵入的自动 2PC

核心思路：拦截 SQL，自动生成回滚日志

┌──────────────────────────────────────────────────────────┐
│                    Seata AT 执行流程                       │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  一阶段（Branch 注册 + 本地事务提交）                       │
│  ┌─────────────────────────────────────────────┐         │
│  │ 1. 拦截 SQL，查询修改前数据（Before Image）   │         │
│  │ 2. 执行业务 SQL                              │         │
│  │ 3. 查询修改后数据（After Image）              │         │
│  │ 4. 生成 Undo Log 并保存到 undo_log 表         │         │
│  │ 5. 提交本地事务（业务数据 + Undo Log 一起提交）│         │
│  │ 6. 向 TC 注册分支事务，上报状态                │         │
│  └─────────────────────────────────────────────┘         │
│                                                           │
│  二阶段（提交 / 回滚）                                     │
│  ┌─────────────────────────────────────────────┐         │
│  │ 提交：异步删除 Undo Log（因为已提交）          │         │
│  │ 回滚：读取 Undo Log，反向补偿（生成反向 SQL）  │         │
│  └─────────────────────────────────────────────┘         │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 6.2 undo_log 表结构

```sql
-- 每个参与分布式事务的数据库都需要这张表
CREATE TABLE `undo_log` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `branch_id` bigint(20) NOT NULL COMMENT '分支事务ID',
  `xid` varchar(100) NOT NULL COMMENT '全局事务ID',
  `context` varchar(128) NOT NULL COMMENT '上下文',
  `rollback_info` longblob NOT NULL COMMENT '回滚数据（Before/After Image）',
  `log_status` int(11) NOT NULL COMMENT '状态',
  `log_created` datetime NOT NULL COMMENT '创建时间',
  `log_modified` datetime NOT NULL COMMENT '修改时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `ux_undo_log` (`xid`, `branch_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;
```

```java
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private InventoryClient inventoryClient;
    @Autowired
    private PaymentClient paymentClient;

    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public Order createOrder(OrderRequest request) {
        // 1. 创建订单（本地事务，自动记录 Undo Log）
        Order order = new Order()
            .setProductId(request.getProductId())
            .setQuantity(request.getQuantity())
            .setAmount(request.getAmount())
            .setStatus("CREATED");
        orderMapper.insert(order);

        // 2. 扣减库存（远程调用，自动注册分支事务）
        inventoryClient.deduct(request.getProductId(), request.getQuantity());

        // 3. 扣减余额（远程调用，自动注册分支事务）
        paymentClient.pay(order.getId(), request.getAmount());

        // 任一环节异常，Seata 自动回滚所有分支
        return order;
    }
}
```

### 6.4 全局锁与写隔离

```
┌──────────────────────────────────────────────────────────────┐
│                    Seata 全局锁机制                            │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  全局事务 tx-1 扣减库存 product_id=100                        │
│  ┌───────────────────────────────────────────────┐           │
│  │ 1. 一阶段本地提交后，向 TC 申请全局锁            │           │
│  │    锁键: inventory:product_id=100              │           │
│  │ 2. 全局锁持有期间，其他全局事务无法修改同一行     │           │
│  │ 3. tx-1 提交/回滚后释放全局锁                   │           │
│  └───────────────────────────────────────────────┘           │
│                                                               │
│  非全局事务（本地事务）如何处理？                               │
│  ┌───────────────────────────────────────────────┐           │
│  │ 默认：本地事务不受全局锁限制，可能脏写           │           │
│  │ 解决：配置全局锁检查                             │           │
│  │   seata.client.undo.log-serialization=jackson  │           │
│  │   seata.client.undo.only-care-update-columns=true│          │
│  │ 回滚时校验 After Image，若不一致则告警           │           │
│  └───────────────────────────────────────────────┘           │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

**代码实现**：

```java
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private MessageMapper messageMapper;
    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    @Transactional
    public void createOrder(Order order) {
        // 1. 业务操作
        orderMapper.insert(order);

        // 2. 写入本地消息表（同一事务）
        MessageRecord message = new MessageRecord()
            .setTopic("order-created")
            .setKey(order.getId().toString())
            .setBody(JSON.toJSONString(order))
            .setStatus("NEW")
            .setRetryCount(0)
            .setNextRetryTime(LocalDateTime.now().plusSeconds(5));
        messageMapper.insert(message);
    }
}

// 定时任务：扫描未发送的消息
@Component
public class MessageScheduler {

    @Autowired
    private MessageMapper messageMapper;
    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    @Scheduled(fixedDelay = 5000)
    public void sendPendingMessages() {
        List<MessageRecord> messages = messageMapper.selectPending();
        for (MessageRecord msg : messages) {
            try {
                SendResult result = rocketMQTemplate.syncSend(
                    msg.getTopic(), msg.getBody()
                );
                if (result.getSendStatus() == SendStatus.SEND_OK) {
                    messageMapper.updateStatus(msg.getId(), "SENT");
                }
            } catch (Exception e) {
                // 更新重试次数
                messageMapper.incrementRetry(msg.getId());
                if (msg.getRetryCount() >= 5) {
                    messageMapper.updateStatus(msg.getId(), "FAILED");
                    // 告警人工介入
                }
            }
        }
    }
}
```

### 7.2 RocketMQ 事务消息

```
┌──────────────────────────────────────────────────────────────┐
│                RocketMQ 事务消息流程                           │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Producer                        Broker              Consumer │
│     │                              │                    │     │
│     │──1. 发送半消息(Half)────────▶│                    │     │
│     │◀─2. 半消息存储成功──────────│                    │     │
│     │                              │                    │     │
│     │──3. 执行本地事务──┐          │                    │     │
│     │                   │          │                    │     │
│     │    ┌──────────────┘          │                    │     │
│     │    │                          │                    │     │
│     │    ├── 成功 ──4a. Commit────▶│                    │     │
│     │    │                         │──5. 投递消息──────▶│     │
│     │    │                         │                    │     │
│     │    └── 失败 ──4b. Rollback─▶│ (删除半消息)       │     │
│     │                              │                    │     │
│     │  [异常: 未发送4a/4b]         │                    │     │
│     │◀────6. 回查本地事务状态──────│                    │     │
│     │────7. 返回 Commit/Rollback─▶│                    │     │
│     │                              │                    │     │
└──────────────────────────────────────────────────────────────┘
```

**代码实现**：

```java
@Service
public class OrderService {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    @Autowired
    private OrderMapper orderMapper;

    public void createOrder(Order order) {
        // 发送事务消息
        rocketMQTemplate.sendMessageInTransaction(
            "order-created-topic",
            MessageBuilder.withPayload(order)
                .setHeader("orderId", order.getId())
                .build(),
            order  // 传递给本地事务方法的参数
        );
    }
}

@RocketMQTransactionListener
public class OrderTransactionListener implements RocketMQLocalTransactionListener {

    @Autowired
    private OrderMapper orderMapper;

    /**
     * 执行本地事务
     */
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            Order order = (Order) arg;
            orderMapper.insert(order);
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }

    /**
     * 回查本地事务状态
     */
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        String orderId = (String) msg.getHeaders().get("orderId");
        Order order = orderMapper.selectById(orderId);
        if (order != null) {
            return RocketMQLocalTransactionState.COMMIT;
        }
        return RocketMQLocalTransactionState.ROLLBACK;
    }
}
```

## 八、方案选型指南

### 8.1 决策树

```
需要分布式事务？
│
├── 是否允许短暂不一致？
│   ├── NO（强一致）
│   │   ├── 是否涉及资金？
│   │   │   ├── YES ──▶ TCC（精确控制，Try预留资源）
│   │   │   └── NO
│   │   │       ├── 是否能接受 SQL 自动回滚？
│   │   │       │   ├── YES ──▶ Seata AT（低侵入）
│   │   │       │   └── NO  ──▶ TCC
│   │   │       └── 是否已有老系统？
│   │   │           ├── YES ──▶ XA/2PC（无代码侵入）
│   │   │           └── NO  ──▶ Seata AT
│   │   │
│   └── YES（最终一致）
│       ├── 是否有 MQ 基础设施？
│       │   ├── YES ──▶ RocketMQ 事务消息
│       │   └── NO  ──▶ 本地消息表
│       └── 流程是否很长（>5步）？
│           ├── YES ──▶ Saga（编排长流程）
```

### 8.2 实际项目选型参考

| 业务场景 | 推荐方案 | 理由 |
|----------|----------|------|
| 电商下单（库存+支付） | TCC | 资金相关，不能超卖 |
| 订单超时取消 | MQ 延迟消息 + 最终一致 | 允许短暂不一致 |
| 跨行转账 | TCC | 资金安全第一 |
| 优惠券发放 | 本地消息表 | 异步场景，最终一致可接受 |
| 供应链长流程 | Saga | 流程长，需要补偿编排 |
| 内部系统跨库查询 | Seata AT | 低侵入，自动回滚 |
| 日志/监控数据同步 | MQ 事务消息 | 数据量大，最终一致即可 |

## 九、面试高频问题速查

### 9.1 基础概念

| 问题 | 核心答案 |
|------|----------|
| 分布式事务和本地事务的区别？ | 本地事务通过 DB 保障 ACID；分布式事务跨服务/DB，需要额外机制保障一致性 |
| CAP 为什么不能同时满足？ | 网络分区必然存在（P），CP 牺牲可用性，AP 牺牲一致性 |
| BASE 理论的核心？ | 基本可用 + 软状态 + 最终一致性，是分布式事务的理论基础 |
| 强一致和最终一致怎么选？ | 资金/库存等核心链路强一致，日志/通知等辅助链路最终一致 |

### 9.2 方案对比

| 问题 | 核心答案 |
|------|----------|
| 2PC 和 3PC 的区别？ | 3PC 增加CanCommit 阶段 + 超时机制，但仍无法解决网络分区下的不一致 |
| TCC 的空回滚和悬挂？ | 空回滚：Try未执行Cancel被调用→检查Try记录；悬挂：Cancel先于Try→Try前检查Cancel记录 |
| Saga 的两种编排方式？ | Choreography 事件驱动（松耦合难追踪）vs Orchestration 中心协调（集中管控单点风险） |
| Seata AT 的全局锁？ | 一阶段本地提交后持有全局锁，防止其他全局事务同时修改同一行 |
| 本地消息表和 MQ 事务消息的区别？ | 本地消息表需要额外表+定时任务；MQ事务消息由Broker保证，更简洁 |

### 9.3 深度问题

| 问题 | 核心答案 |
|------|----------|
| Seata AT 回滚时 After Image 校验失败怎么办？ | 说明被本地事务脏写，默认回滚失败告警人工介入 |
| TCC 的 Confirm/Cancel 为什么要幂等？ | 网络超时重试可能重复调用，不幂等会导致重复扣减/释放 |
| 分布式事务和分布式锁的关系？ | 分布式锁保证互斥访问，分布式事务保证多操作一致性；两者常常配合使用 |
| 如何保证消息消费的幂等？ | 消费端通过业务唯一ID去重（数据库唯一键/Redis SET/状态机判断） |
| 大规模系统如何减少分布式事务？ | 合理拆分服务边界，将强一致操作聚合在同一服务内，跨服务用最终一致 |

## 十、总结

```
┌──────────────────────────────────────────────────────────────┐
│                   分布式事务核心要点                           │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. 选型原则：                                                │
│     能不用分布式事务就不用 → 业务上避免跨服务强一致             │
│     必须用时，强一致选 TCC/AT，最终一致选消息                  │
│                                                               │
│  2. 兜底思维：                                                │
│     任何方案都可能出现异常，必须有补偿和对账机制                │
│     幂等是分布式事务的基石                                     │
│                                                               │
│  3. 性能考量：                                                │
│     强一致 = 加锁 = 低吞吐，最终一致 = 异步 = 高吞吐           │
│     核心链路强一致，非核心链路最终一致                          │
│                                                               │
│  4. 工程实践：                                                │
│     Seata 是目前最成熟的 Java 分布式事务框架                   │
│     AT 模式适合大多数场景，TCC 适合资金类精确控制               │
│     生产环境必须配套监控、告警、对账                            │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```
