## 一、CAP 与 BASE

### 1.1 CAP 定理

```
分布式系统最多同时满足三个中的两个：

        ┌─────────────┐
        │ Consistency │ 一致性
        └──────┬──────┘
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌──────────┐
│   CP   │ │   CA   │ │   AP     │
│ZooKeeper│ │ RDBMS │ │ Cassandra│
│强一致   │ │单机    │ │高可用    │
│可用性低 │ │不分区  │ │最终一致  │
└────────┘ └────────┘ └──────────┘

实际选择：P（分区容错）必须满足
  ├── 金融/库存 → CP（宁可不可用也不能数据错）
  └── 社交/电商 → AP（短暂不一致可接受，优先可用）
```

### 1.2 BASE 理论

| 原则 | 含义 |
|------|------|
| Basically Available | 基本可用（允许响应变慢或功能降级） |
| Soft State | 软状态（允许中间状态） |
| Eventually Consistent | 最终一致性（一段时间后达到一致） |

## 二、一致性算法

### 2.1 Raft

```
┌──────────────────────────────────────────────────────────┐
│                  Raft 算法核心                            │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  角色：Leader / Follower / Candidate                      │
│                                                           │
│  Leader 选举：                                            │
│  1. Follower 超时未收到心跳 → 变为 Candidate              │
│  2. Candidate 投自己一票 + 请求其他节点投票               │
│  3. 获得多数票（N/2 + 1）→ 成为 Leader                   │
│  4. Leader 周期性发送心跳维持权威                         │
│                                                           │
│  日志复制：                                               │
│  1. 客户端请求 → Leader 写入本地日志                      │
│  2. Leader 将日志复制到所有 Follower                      │
│  3. 多数节点确认 → Leader 提交日志                       │
│  4. 通知客户端成功                                        │
│                                                           │
│  关键：多数派（Majority）保证一致性                       │
│  3 节点容忍 1 节点故障                                    │
│  5 节点容忍 2 节点故障                                    │
│                                                           │
│  应用：etcd / Consul / Nacos(CP模式) / TiKV              │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 2.2 Paxos vs Raft

| 特性 | Paxos | Raft |
|------|-------|------|
| 理解难度 | 极难 | 较易 |
| 角色 | Proposer/Acceptor/Learner | Leader/Follower/Candidate |
| Leader | 不强制选主 | 强制选主 |
| 工程实现 | 困难（Google Chubby 作者都觉得难） | 简单（etcd/Consul） |

## 三、分布式 ID

### 3.1 方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| UUID | 随机生成 | 简单 | 无序、占空间、索引效率低 |
| 数据库自增 | auto_increment | 简单有序 | 单点瓶颈、性能差 |
| 号段模式 | 批量取号段 | 减少DB访问 | 需要中间件 |
| 雪花算法 | 时间+机器+序列 | 有序、高性能 | 时钟回拨问题 |
| Redis | INCR | 高性能 | 依赖 Redis |

### 3.2 雪花算法（Snowflake）

```
┌──────────────────────────────────────────────────────────┐
│              Snowflake ID 结构（64 bit）                  │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  0 | 00000000 00000000 00000000 00000000 00000000 0      │
│    | ├──────── 41 bit 时间戳 ────────┤├─10bit─┤├─12bit─┤│
│    |                                       机器ID  序列号 │
│                                                           │
│  1 bit   符号位（永远0）                                  │
│  41 bit  时间戳（毫秒级，约69年）                         │
│  10 bit  机器ID（0~1023，5位数据中心+5位机器）             │
│  12 bit  序列号（0~4095，每毫秒4096个ID）                 │
│                                                           │
│  每毫秒可生成 4096 个 ID                                  │
│  单机每秒可生成 400万+ ID                                 │
│                                                           │
│  时钟回拨问题：                                           │
│  ├── 等待回拨时间过去                                     │
│  ├── 使用历史最大时间戳                                   │
│  └── 报错拒绝生成（Leaf 的方案）                          │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

## 四、分布式锁

### 4.1 方案对比

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| Redis | SETNX + 过期 + Lua | 高性能 | 主从切换可能丢锁 |
| ZooKeeper | 临时顺序节点 | 强一致 | 性能低 |
| MySQL | 行锁/唯一索引 | 简单 | 性能差 |
| etcd | Lease + Revision | 强一致 | 生态较新 |

### 4.2 RedLock

```
┌──────────────────────────────────────────────────────────┐
│                  RedLock 算法                              │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  前提：5 个独立 Redis Master                              │
│                                                           │
│  1. 获取当前时间 T1                                       │
│  2. 依次向 5 个实例加锁（SETNX + 超时）                  │
│  3. 计算加锁耗时 = T2 - T1                                │
│  4. 成功条件：                                            │
│     ├── 多数实例加锁成功（≥3）                            │
│     └── 加锁耗时 < 锁超时时间                             │
│  5. 失败则向所有实例释放锁                                │
│                                                           │
│  争议：                                                   │
│  ├── Martin Kleppmann 认为有 GC/网络延迟导致不安全        │
│  ├── Antirez（Redis 作者）反驳认为实际足够安全            │
│  └── 实际：大多数场景 Redisson 单节点锁已够用             │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

## 五、服务治理

### 5.1 限流

```
┌──────────────────────────────────────────────────────────┐
│                    限流算法                                │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  1. 固定窗口计数器                                        │
│     └── 每分钟100次 → 临界点可能突增到200次              │
│                                                           │
│  2. 滑动窗口计数器                                        │
│     └── 细粒度窗口滑动，解决临界问题                      │
│                                                           │
│  3. 漏桶算法                                              │
│     ┌─────────┐    ┌───┐                                 │
│     │ 请求涌入 │───▶│漏 │───▶ 匀速流出                    │
│     │（任意速率）│  │桶 │     （固定速率）                 │
│     └─────────┘    └───┘                                 │
│     → 平滑流量，但无法应对突发                            │
│                                                           │
│  4. 令牌桶算法（推荐）                                    │
│     ┌───┐         ┌─────────┐                            │
│     │令牌│──匀速──▶│  桶     │───▶ 请求消费令牌           │
│     │生成│  放入   │(有容量) │     桶空则拒绝              │
│     └───┘         └─────────┘                            │
│     → 允许突发流量（桶中有积累令牌）                      │
│     → Guava RateLimiter / Sentinel 实现                   │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 5.2 熔断降级

```
┌──────────────────────────────────────────────────────────┐
│                  熔断器模式（Circuit Breaker）            │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  ┌──────────┐   错误率超阈值   ┌──────────┐              │
│  │  CLOSED  │──────────────▶  │   OPEN   │              │
│  │  (正常)   │◀────────────── │  (熔断)   │              │
│  └──────────┘   探测成功      └─────┬────┘              │
│       ▲                            │                     │
│       │                            │ 超时后              │
│       │                            ▼                     │
│       │                      ┌──────────┐               │
│       └────────────────────  │HALF-OPEN │               │
│           探测成功            │  (半开)   │               │
│                                └──────────┘               │
│                                                           │
│  降级策略：                                               │
│  ├── 返回默认值（兜底数据）                               │
│  ├── 返回缓存数据                                         │
│  ├── 写入队列后续处理                                     │
│  └── 转发到备用服务                                       │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 5.3 服务降级层次

```
┌────────────────────────────────────────────────────────┐
│                 降级层次（从外到内）                     │
├────────────────────────────────────────────────────────┤
│                                                         │
│  1. 页面降级      静态页面替换动态页面                   │
│  2. 页面片段降级   推荐模块/评论模块隐藏                │
│  3. 服务接口降级   非核心接口返回兜底数据                │
│  4. 读服务降级     读缓存 / 读从库                      │
│  5. 写服务降级     先写缓存异步落库                     │
│                                                         │
│  降级开关：                                             │
│  ├── 配置中心动态开关（Nacos/Apollo）                   │
│  ├── 数据库开关表                                      │
│  └── Sentinel 规则动态生效                              │
│                                                         │
└────────────────────────────────────────────────────────┘
```

## 六、分布式一致性

### 6.1 一致性模型

| 模型 | 说明 | 典型系统 |
|------|------|----------|
| 强一致性 | 写入后立即可读 | ZooKeeper / etcd |
| 顺序一致性 | 所有节点看到相同顺序 | ZooKeeper |
| 因果一致性 | 有因果关系的操作顺序一致 | |
| 最终一致性 | 一段时间后一致 | DNS / Redis 主从 |
| 弱一致性 | 不保证一致 | CDN 缓存 |

### 6.2 分布式事务方案速查

| 方案 | 一致性 | 原理 | 适用 |
|------|--------|------|------|
| 2PC/XA | 强一致 | 协调者+参与者两阶段 | 传统DB跨库 |
| TCC | 强一致 | Try预留+Confirm+Cancel | 资金类核心 |
| Seata AT | 强一致 | 自动生成UndoLog | 低侵入场景 |
| Saga | 最终一致 | 正向+补偿 | 长流程 |
| MQ事务 | 最终一致 | 半消息+本地事务 | 异步解耦 |

## 七、高频面试题

| 问题 | 核心答案 |
|------|----------|
| CAP 怎么选？ | P 必须满足；金融选CP，互联网选AP |
| Raft 怎么选举 Leader？ | Follower超时→Candidate→获得多数票→Leader |
| 雪花算法的时钟回拨怎么处理？ | 等待回拨过去/使用历史最大时间/拒绝生成 |
| 分布式锁用 Redis 还是 ZooKeeper？ | 高性能选Redis；强一致选ZK；大多场景Redis够用 |
| 限流算法有哪些？ | 固定窗口/滑动窗口/漏桶/令牌桶（推荐） |
| 熔断器三种状态？ | Closed(正常) → Open(熔断) → Half-Open(半开探测) |
| 如何设计一个高可用系统？ | 冗余部署+熔断限流降级+超时重试+幂等+监控告警 |
