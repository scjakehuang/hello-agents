## 一、运行时数据区

```
┌──────────────────────────────────────────────────────────────┐
│                    JVM 运行时数据区                            │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  线程私有：                                                    │
│  ┌────────────────────────────────────────────────────┐      │
│  │ 程序计数器（PC Register）                            │      │
│  │ 当前线程执行的字节码行号，唯一不会 OOM 的区域         │      │
│  └────────────────────────────────────────────────────┘      │
│  ┌────────────────────────────────────────────────────┐      │
│  │ 虚拟机栈（VM Stack）                                │      │
│  │ 每个方法一个栈帧：                                   │      │
│  │ ├── 局部变量表（基本类型 + 对象引用 + returnAddress） │      │
│  │ ├── 操作数栈                                        │      │
│  │ ├── 动态链接（指向运行时常量池的方法引用）            │      │
│  │ └── 方法返回地址                                    │      │
│  │ SOF: 栈深度超限  OOM: 无法分配新栈                   │      │
│  └────────────────────────────────────────────────────┘      │
│  ┌────────────────────────────────────────────────────┐      │
│  │ 本地方法栈（Native Method Stack）                    │      │
│  │ 为 Native 方法服务                                  │      │
│  └────────────────────────────────────────────────────┘      │
│                                                               │
│  线程共享：                                                    │
│  ┌────────────────────────────────────────────────────┐      │
│  │ 堆（Heap）                                          │      │
│  │ ├── 新生代 (1/3)                                    │      │
│  │ │   ├── Eden (8/10)                                 │      │
│  │ │   ├── Survivor0 (1/10)                            │      │
│  │ │   └── Survivor1 (1/10)                            │      │
│  │ └── 老年代 (2/3)                                    │      │
│  │ OOM: Java heap space                                │      │
│  └────────────────────────────────────────────────────┘      │
│  ┌────────────────────────────────────────────────────┐      │
│  │ 方法区（Method Area）/ 元空间（Metaspace）           │      │
│  │ ├── 类信息、常量、静态变量、JIT编译后的代码           │      │
│  │ ├── 运行时常量池（字符串常量池 JDK7+ 移到堆中）       │      │
│  │ └── JDK8+ 用本地内存实现（Metaspace），不再 OOM       │      │
│  └────────────────────────────────────────────────────┘      │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## 二、垃圾回收

### 2.1 对象存活判定

| 算法 | 原理 | 应用 |
|------|------|------|
| 引用计数法 | 引用+1，失引-1，为0则回收 | Python；无法解决循环引用 |
| 可达性分析 | GC Roots 出发，不可达则回收 | Java 采用 |

**GC Roots 包含**：

```
GC Roots
├── 虚拟机栈中引用的对象（局部变量）
├── 方法区中静态属性引用的对象
├── 方法区中常量引用的对象
├── 本地方法栈中 JNI 引用的对象
├── JVM 内部引用（基本类型对应的 Class 对象等）
└── 同步锁（synchronized）持有的对象
```

### 2.2 垃圾回收算法

```
┌──────────────────────────────────────────────────────────────┐
│                    三种核心回收算法                             │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. 标记-清除（Mark-Sweep）                                   │
│     ┌──┬──┬──┬──┬──┬──┬──┬──┐                               │
│     │A │  │B │  │C │  │D │  │                               │
│     └──┘  └──┘  └──┘  └──┘                                   │
│     缺点：内存碎片化                                           │
│                                                               │
│  2. 标记-复制（Copying）                                      │
│     ┌──┬──┬──┬──┐    ┌──┬──┬──┬──┐                          │
│     │A │  │B │  │ →  │A │B │  │  │                          │
│     └──┴──┴──┴──┘    └──┴──┴──┴──┘                          │
│     新生代用，Eden+Survivor 复制到另一个 Survivor              │
│     缺点：空间利用率 50%                                       │
│                                                               │
│  3. 标记-整理（Mark-Compact）                                 │
│     ┌──┬──┬──┬──┬──┬──┬──┬──┐                               │
│     │A │  │B │  │C │  │  │  │                               │
│     └──┴──┴──┴──┴──┴──┴──┴──┘                               │
│               ↓ 整理后                                         │
│     ┌──┬──┬──┬──┬──┬──┬──┬──┐                               │
│     │A │B │C │  │  │  │  │  │                               │
│     └──┴──┴──┴──┴──┴──┴──┴──┘                               │
│     老年代用，无碎片但移动开销大                                │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 分代回收

```
┌──────────────────────────────────────────────────────────┐
│                    分代回收模型                             │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  新生代（Minor GC / Young GC）                            │
│  ┌──────────────────────────────────────────┐            │
│  │           Eden    S0        S1            │            │
│  │         ┌──────┬──────┬──────┐            │            │
│  │         │ 80%  │ 10%  │ 10%  │            │            │
│  │         └──────┴──────┴──────┘            │            │
│  │                                          │            │
│  │ 新对象分配在 Eden                         │            │
│  │ Eden 满 → Minor GC                       │            │
│  │ 存活对象复制到 S0/S1，年龄+1              │            │
│  │ 年龄 ≥ 15（默认）→ 晋升老年代             │            │
│  │ 大对象直接进老年代                         │            │
│  └──────────────────────────────────────────┘            │
│                                                           │
│  老年代（Major GC / Full GC）                             │
│  ┌──────────────────────────────────────────┐            │
│  │ 长期存活的对象、大对象                     │            │
│  │ 空间不足 → Major GC / Full GC             │            │
│  │ Stop-The-World 时间更长                    │            │
│  └──────────────────────────────────────────┘            │
│                                                           │
│  GC 触发条件：                                            │
│  Minor GC: Eden 区满                                      │
│  Full GC:  老年代满 / Metaspace 满 / 显式System.gc()      │
│            / 新生代晋升老年代空间不足                       │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 2.4 垃圾回收器

```
┌──────────────────────────────────────────────────────────────┐
│                    垃圾回收器全景                              │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  新生代              老年代              组合                  │
│  ┌──────────┐      ┌──────────┐                              │
│  │ Serial   │ ───▶ │ Serial   │      单线程，Client 模式      │
│  │  复制     │      │ Old(MSC) │                              │
│  └──────────┘      └──────────┘                              │
│                                                               │
│  ┌──────────┐      ┌──────────┐                              │
│  │ ParNew   │ ───▶ │ CMS      │      并发标记清除，停顿短      │
│  │  复制     │      │ 标记清除  │      JDK9 标记废弃           │
│  └──────────┘      └──────────┘                              │
│                                                               │
│  ┌──────────┐      ┌──────────┐                              │
│  │Parallel  │ ───▶ │Parallel  │      吞吐量优先              │
│  │ Scavenge │      │ Old(MSC) │      适合后台计算             │
│  └──────────┘      └──────────┘                              │
│                                                               │
│  ┌─────────────────────────────────┐                         │
│  │          G1 (Garbage First)      │  分 Region，停顿可预测   │
│  │          JDK9 默认               │  兼顾吞吐和延迟          │
│  └─────────────────────────────────┘                         │
│                                                               │
│  ┌─────────────────────────────────┐                         │
│  │          ZGC                     │  超低延迟 < 1ms         │
│  │          JDK11+                  │  读屏障 + 染色指针       │
│  └─────────────────────────────────┘                         │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 2.5 G1 核心原理

```
┌──────────────────────────────────────────────────────────────┐
│                    G1 回收器核心设计                           │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Region 分区（1~32MB，默认 2048 个）                          │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┐                  │
│  │ E  │ S  │ O  │ O  │ H  │ E  │ O  │ E  │                  │
│  └────┴────┴────┴────┴────┴────┴────┴────┘                  │
│  E=Eden  S=Survivor  O=Old  H=Humongous(大对象)              │
│                                                               │
│  回收策略：                                                    │
│  1. 维护每个 Region 的回收价值（垃圾占比）                     │
│  2. 优先回收价值高的 Region（Garbage First）                  │
│  3. 用户设定期望停顿时间（-XX:MaxGCPauseMillis）              │
│  4. 根据停顿目标选择回收哪些 Region                            │
│                                                               │
│  三阶段：                                                     │
│  ├── Young GC：回收 Eden + Survivor Region                   │
│  ├── 并发标记：类似 CMS，但基于 Region                       │
│  └── Mixed GC：回收部分老年代 Region + 全部新生代             │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## 三、类加载

### 3.1 类加载过程

```
┌───────────────────────────────────────────────────────────┐
│                    类加载生命周期                            │
├───────────────────────────────────────────────────────────┤
│                                                            │
│  加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载        │
│  |________链接________|                                     │
│                                                            │
│  加载：                                                     │
│  ├── 通过全限名获取二进制字节流                              │
│  ├── 将字节流转化为方法区的运行时数据结构                    │
│  └── 在堆中生成 java.lang.Class 对象                       │
│                                                            │
│  验证：文件格式、元数据、字节码、符号引用验证                │
│                                                            │
│  准备：为静态变量分配内存并赋零值（final static 在此赋值）   │
│                                                            │
│  解析：将常量池符号引用替换为直接引用                        │
│                                                            │
│  初始化：执行 <clinit>() 方法（静态变量赋值 + static 块）   │
│                                                            │
└───────────────────────────────────────────────────────────┘
```

### 3.2 双亲委派模型

```
┌──────────────────────────────────────────────────────────┐
│                  双亲委派模型                              │
├──────────────────────────────────────────────────────────┤
│                                                           │
│          ┌────────────────────┐                           │
│          │ Bootstrap ClassLdr │  rt.jar, tools.jar       │
│          │ (C++实现)          │                           │
│          └─────────┬──────────┘                           │
│                    │ parent                               │
│          ┌─────────┴──────────┐                           │
│          │ Extension ClassLdr │  ext/*.jar               │
│          └─────────┬──────────┘                           │
│                    │ parent                               │
│          ┌─────────┴──────────┐                           │
│          │ Application ClassLdr│ classpath               │
│          └─────────┬──────────┘                           │
│                    │ parent                               │
│          ┌─────────┴──────────┐                           │
│          │ Custom ClassLoader │  自定义加载路径            │
│          └────────────────────┘                           │
│                                                           │
│  委派流程：子加载器先委托父加载器加载                       │
│  好处：避免核心类被篡改（如自定义 java.lang.String）       │
│                                                           │
│  打破双亲委派的场景：                                      │
│  ├── SPI 机制（JDBC）：线程上下文类加载器                   │
│  ├── Tomcat：每个 WebApp 独立 ClassLoader                  │
│  ├── OSGi：网状加载                                       │
│  └── 热部署：替换 ClassLoader 重新加载类                   │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

## 四、JVM 调优

### 4.1 常用参数

```
# 堆大小
-Xms4g                          # 初始堆大小
-Xmx4g                          # 最大堆大小（生产 Xms=Xmx，避免动态扩缩）
-Xmn2g                          # 新生代大小
-Xss512k                        # 线程栈大小

# 元空间
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=512m

# GC 选择
-XX:+UseG1GC                    # 使用 G1
-XX:MaxGCPauseMillis=200        # G1 目标停顿时间

# GC 日志（JDK8）
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/tmp/gc.log

# GC 日志（JDK9+）
-Xlog:gc*:file=/tmp/gc.log:time,uptime,level,tags

# OOM 时导出堆
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/dump.hprof
```

### 4.2 常见 OOM 类型

| OOM 类型 | 原因 | 排查 |
|----------|------|------|
| Java heap space | 堆内存不足，对象太多 | jmap -histo / MAT 分析 |
| Metaspace | 类加载过多 | 检查动态代理/CGLIB |
| GC overhead limit | GC 回收效率低 | 检查内存泄漏 |
| Direct buffer memory | 堆外内存不足 | 检查 NIO/Netty |
| unable to create new native thread | 线程数超限 | 减少线程数或增大 OS 限制 |

## 五、高频面试题

| 问题 | 核心答案 |
|------|----------|
| JVM 内存模型？ | 线程私有：PC/栈/本地栈；共享：堆/方法区 |
| 对象一定在堆上分配吗？ | 不一定，JIT 逃逸分析后可能栈上分配（标量替换） |
| CMS 的缺点？ | 浮动垃圾、内存碎片、CPU 敏感、Concurrent Mode Failure |
| G1 和 CMS 的区别？ | G1 分 Region、可预测停顿、整体标记-整理局部复制 |
| 什么情况触发 Full GC？ | 老年代满、Metaspace 满、System.gc()、晋升失败 |
| 如何排查 OOM？ | -XX:+HeapDumpOnOutOfMemoryError → MAT 分析 → 定位大对象/泄漏引用链 |
| 双亲委派模型是什么？ | 子加载器先委托父加载器，保证核心类安全不被篡改 |
| 逃逸分析优化了什么？ | 栈上分配（减少GC）、标量替换、锁消除 |
