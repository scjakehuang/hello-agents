## 一、Java基础

### 1. 引用类型

| 引用类型 | 说明 | 使用场景 |
|----------|------|----------|
| **强引用** | 引用存在时不能被垃圾回收 | 普通对象引用 |
| **软引用** | `SoftReference`，空间不够时被回收 | 缓存 |
| **弱引用** | `WeakReference`，GC时就会被回收 | ThreadLocal中避免内存泄漏 |
| **虚引用** | `PhantomReference`，回收时放入queue | 管理堆外内存（NIO Zero Copy） |

**示例代码：**
```java
// 软引用
SoftReference<Object> softRef = new SoftReference<>(new Object());

// 弱引用 - ThreadLocal中使用
WeakReference<Object> weakRef = new WeakReference<>(new Object());

// 虚引用 - 管理直接内存
PhantomReference<Object> phantomRef = new PhantomReference<>(new Object(), queue);

// NIO直接内存
ByteBuffer buffer = ByteBuffer.allocateDirect(1024);
```

### 3. 锁机制

#### 3.1 Java锁机制概述

| 锁类型 | 说明 |
|--------|------|
| 乐观锁 | CAS实现，适合读多写少 |
| 悲观锁 | synchronized、ReentrantLock |
| 公平锁 | 按等待顺序获取锁 |
| 非公平锁 | 允许插队，性能更高 |
| 可重入锁 | 同一线程可重复获取锁 |
| 读写锁 | 读读共享，写写/读写互斥 |

#### 3.2 ReentrantLock底层实现

```
ReentrantLock
    └── Sync (继承AQS)
        ├── FairSync (公平锁)
        └── NonfairSync (非公平锁)
```

**核心组件：**
- **AQS (AbstractQueuedSynchronizer)**
  - state：锁状态（0未持有，>0持有次数）
  - CLH队列：等待线程的双向链表
  - CAS操作：state的原子更新

#### 3.3 AQS原理

```
┌─────────────────────────────────────────┐
│              AQS 核心结构                │
├─────────────────────────────────────────┤
│  state (volatile int)                   │
│  ├─ 0: 锁未被持有                        │
│  └─ >0: 锁被持有，值为重入次数            │
├─────────────────────────────────────────┤
│  CLH 队列 (双向链表)                     │
│  head → Node → Node → Node → tail       │
│         ↓      ↓      ↓                 │
│       Thread Thread Thread              │
├─────────────────────────────────────────┤
│  核心方法                                │
│  ├─ tryAcquire(): 尝试获取锁             │
│  ├─ tryRelease(): 尝试释放锁             │
│  ├─ acquireQueued(): 队列中获取锁        │
│  └─ release(): 释放锁并唤醒后继节点       │
└─────────────────────────────────────────┘
```

#### 3.4 公平锁 vs 非公平锁

| 对比项 | 公平锁 | 非公平锁 |
|--------|--------|----------|
| 获取方式 | 先来先得，排队获取 | 可插队，尝试直接获取 |
| 性能 | 较低（需要维护队列） | 较高 |
| 实现 | `FairSync` | `NonfairSync` |
| 适用场景 | 要求公平性 | 追求性能 |

### 4. 线程池

#### 4.1 核心参数

```java
public ThreadPoolExecutor(
    int corePoolSize,      // 核心线程数
    int maximumPoolSize,   // 最大线程数
    long keepAliveTime,    // 非核心线程空闲存活时间
    TimeUnit unit,         // 时间单位
    BlockingQueue<Runnable> workQueue,  // 工作队列
    ThreadFactory threadFactory,        // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)
```

**参数意义：**

| 参数 | 说明 |
|------|------|
| corePoolSize | 核心线程数，即使空闲也不会被回收 |
| maximumPoolSize | 最大线程数，队列满时创建非核心线程 |
| keepAliveTime | 非核心线程空闲存活时间 |
| workQueue | 等待执行的任务队列 |
| handler | 拒绝策略（AbortPolicy、CallerRunsPolicy等） |

#### 4.2 常用线程池

| 线程池 | 特点 | 适用场景 |
|--------|------|----------|
| FixedThreadPool | 固定线程数，无界队列 | 负载较重的服务器 |
| CachedThreadPool | 可缓存线程池，无限大 | 短期异步任务 |
| SingleThreadExecutor | 单线程 | 顺序执行任务 |
| ScheduledThreadPool | 定时线程池 | 定时任务 |

#### 4.3 执行流程

```
任务提交
    │
    ▼
核心线程数未满？
    ├─ 是 → 创建核心线程执行
    └─ 否 ↓
            工作队列未满？
            ├─ 是 → 加入队列等待
            └─ 否 ↓
                    最大线程数未满？
                    ├─ 是 → 创建非核心线程执行
                    └─ 否 → 执行拒绝策略
```

### 5. 读写锁

**实现原理：**
- 使用AQS的state的高16位表示读锁，低16位表示写锁
- 读锁：共享锁，多个线程可同时读
- 写锁：独占锁，写时阻塞所有读和写

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock readLock = rwLock.readLock();   // 读锁
Lock writeLock = rwLock.writeLock(); // 写锁
```

### 6. HashMap与ConcurrentHashMap

#### 6.1 HashMap多线程问题

| 问题 | 说明 |
|------|------|
| 数据丢失 | 多线程put导致覆盖 |
| 死循环(JDK7) | 扩容时链表形成环 |
| 数据不一致 | 并发读写可见性问题 |

#### 6.2 ConcurrentHashMap原理对比

| 版本 | 实现方式 | 特点 |
|------|----------|------|
| **JDK7** | Segment分段锁 | 16个Segment，每个Segment继承ReentrantLock |
| **JDK8** | CAS + synchronized | 锁粒度更细，锁单个桶 |

**JDK8优化原因：**
1. 减少内存占用（无Segment对象）
2. 锁粒度更细，并发度更高
3. 使用CAS减少锁竞争

#### 6.3 ConcurrentHashMap JDK8结构

```
ConcurrentHashMap
├── Node<K,V>[] table  // 桶数组
├── 每个桶
│   ├── Node (链表节点)
│   └── TreeNode (红黑树节点，链表长度>8时转换)
└── 关键操作
    ├── put(): CAS + synchronized
    ├── get(): 无锁，volatile保证可见性
    └── 扩容: 多线程协助扩容
```

### 7. JVM垃圾回收

#### 7.1 垃圾标记算法

| 算法 | 说明 | 优缺点 |
|------|------|--------|
| 引用计数法 | 引用为0则回收 | 无法处理循环引用 |
| 可达性分析 | 从GC Roots出发遍历 | 主流方案 |

**GC Roots：**
- 虚拟机栈中的引用
- 方法区类静态属性引用
- 方法区常量池引用
- 本地方法栈JNI引用

#### 7.2 JVM内存划分

```
┌─────────────────────────────────────────────┐
│                  JVM 内存区域                │
├─────────────────────────────────────────────┤
│  堆 (Heap)                                  │
│  ├── 新生代 (Young Generation)              │
│  │   ├── Eden区                             │
│  │   ├── Survivor 0 (From)                  │
│  │   └── Survivor 1 (To)                    │
│  └── 老年代 (Old Generation)                │
├─────────────────────────────────────────────┤
│  方法区 (Method Area / Metaspace)           │
├─────────────────────────────────────────────┤
│  虚拟机栈 (VM Stack)                         │
├─────────────────────────────────────────────┤
│  本地方法栈 (Native Method Stack)           │
├─────────────────────────────────────────────┤
│  程序计数器 (Program Counter Register)      │
└─────────────────────────────────────────────┘
```

**为什么要有新生代和老年代：**
- 新生代：对象生命周期短，存活率低，适合复制算法
- 老年代：对象生命周期长，存活率高，适合标记-整理算法

#### 7.3 新生代回收

**Minor GC流程：**
1. Eden满时触发
2. 存活对象复制到Survivor
3. Survivor中年龄达到阈值晋升老年代
4. 动态年龄判定：Survivor中相同年龄对象总和>Survivor空间一半

#### 7.4 CMS收集器

**回收过程：**

```
1. 初始标记 (STW) → 标记GC Roots直接关联的对象
2. 并发标记       → 与用户线程并发，遍历对象图
3. 重新标记 (STW) → 修正并发标记期间变动的对象
4. 并发清除       → 与用户线程并发，清除标记对象
```

**CMS缺点与解决方案：**

| 缺点 | 解决方案 |
|------|----------|
| CPU敏感 | 增加CPU资源或调整CMS线程数 |
| 浮动垃圾 | 提前触发CMS（-XX:CMSInitiatingOccupancyFraction） |
| 内存碎片 | 开启压缩（-XX:+UseCMSCompactAtFullCollection） |
| 标记失败 | 降低触发阈值 |

- 写操作：创建新版本，旧版本放入Undo Log

### 3. 索引

#### 3.1 B+树结构

```
为什么使用B+树：
1. 层级少，IO次数少（通常3层可存千万数据）
2. 叶子节点形成链表，范围查询高效
3. 非叶子节点只存索引，单页存更多数据

为什么查询快：
1. 磁盘IO少（树高度低）
2. 数据按顺序存储，范围查询效率高
3. 叶子节点双向链表，支持顺序访问
```

#### 3.2 聚簇索引 vs 非聚簇索引

| 对比项 | 聚簇索引 | 非聚簇索引 |
|--------|----------|------------|
| 叶子节点 | 存储完整数据行 | 存储主键值 |
| 数量 | 一张表只有一个 | 可以有多个 |
| 查询 | 直接返回数据 | 需要回表查询 |
| 存储引擎 | InnoDB主键索引 | InnoDB辅助索引 |

**InnoDB vs MyISAM：**

| 对比项 | InnoDB | MyISAM |
|--------|--------|--------|
| 索引类型 | 聚簇索引 | 非聚簇索引 |
| 叶子节点Data | 完整数据行 | 数据物理地址 |
| 事务 | 支持 | 不支持 |
| 行锁 | 支持 | 不支持 |

#### 3.3 索引失效场景

| 场景 | 说明 |
|------|------|
| `LIKE '%abc'` | 前缀模糊匹配 |
| 对索引列使用函数 | `WHERE YEAR(create_time) = 2024` |
| 隐式类型转换 | 字符串字段用数字查询 |
| `OR`条件 | 其中一列无索引 |
| `!=` 或 `<>` | 不等于操作 |
| `IS NULL` / `IS NOT NULL` | 索引列允许NULL时可能失效 |

#### 3.4 组合索引（联合索引）

**最左前缀原则：**
```sql
-- 索引: (a, b, c)
SELECT * FROM t WHERE a = 1;              -- 命中
SELECT * FROM t WHERE a = 1 AND b = 2;    -- 命中
SELECT * FROM t WHERE b = 2;              -- 未命中
SELECT * FROM t WHERE a = 1 AND c = 3;    -- 部分命中(a)
```

### 4. 事务

#### 4.1 ACID特性

| 特性 | 说明 |
|------|------|
| 原子性 (Atomicity) | 事务不可分割 |
| 一致性 (Consistency) | 事务前后数据一致 |
| 隔离性 (Isolation) | 事务之间互不干扰 |
| 持久性 (Durability) | 事务提交后永久保存 |

#### 4.2 隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|------------|------|
| 读未提交 | ✓ | ✓ | ✓ |
| 读已提交 | ✗ | ✓ | ✓ |
| 可重复读 | ✗ | ✗ | ✓ (InnoDB通过MVCC+间隙锁解决) |
| 串行化 | ✗ | ✗ | ✗ |

### 5. Spring事务传播特性

| 传播行为 | 说明 |
|----------|------|
| REQUIRED | 有事务则加入，无则新建（默认） |
| REQUIRES_NEW | 新建事务，挂起当前事务 |
| SUPPORTS | 有事务则加入，无则非事务运行 |
| NOT_SUPPORTED | 非事务运行，挂起当前事务 |
| NEVER | 非事务运行，有事务则抛异常 |
| MANDATORY | 必须有事务，否则抛异常 |
| NESTED | 嵌套事务 |

### 6. 主从同步

#### 6.1 同步流程

```
主库                        从库
  │                          │
  │  1. 写入Binlog           │
  │─────────────────────────>│
  │                          │  2. I/O线程写入Relay Log
  │                          │
  │                          │  3. SQL线程重放Relay Log
  │                          │
```

#### 6.2 主从延迟解决方案

| 方案 | 说明 |
|------|------|
| 写后读走主库 | 关键业务强制路由主库 |
| 半同步复制 | 等待至少一个从库确认 |
| 并行复制 | 多线程重放Relay Log |
| 业务容忍 | 非关键业务接受短暂延迟 |

## 三、Redis

### 1. 内部数据结构

| 类型 | 底层实现 | 使用场景 |

| Set | HashTable + IntSet | 去重、标签 |
| ZSet | SkipList + HashTable | 排行榜 |

### 2. 单线程模型

**为什么Redis快：**
1. 纯内存操作
2. 单线程避免上下文切换
3. IO多路复用（epoll）
4. 高效的数据结构

### 3. 缓存问题

#### 3.1 缓存穿透

**问题：** 查询不存在的数据，缓存和数据库都无

**解决方案：**
- 布隆过滤器
- 缓存空值（设置短过期时间）

#### 3.2 缓存击穿

**问题：** 热点key过期，大量请求直接打到数据库

**解决方案：**
- 热点数据永不过期
- 互斥锁（只允许一个线程重建缓存）

#### 3.3 缓存雪崩

**问题：** 大量key同时过期

**解决方案：**
- 过期时间随机化
- 多级缓存
- 熔断降级

### 4. 持久化

| 方式 | 说明 | 优缺点 |
|------|------|--------|
| RDB | 快照 | 恢复快，可能丢失数据 |
| AOF | 追加日志 | 数据安全，文件大 |
| 混合 | RDB + AOF | 兼顾两者优点 |

## 四、分布式系统

### 1. CAP理论

| 要素 | 说明 |
|------|------|
| Consistency | 一致性，所有节点数据一致 |
| Availability | 可用性，每个请求都有响应 |
| Partition Tolerance | 分区容错，网络分区时系统仍能运行 |

**取舍：** 分布式系统最多只能同时满足两项

### 2. ZooKeeper vs Eureka

| 对比项 | ZooKeeper | Eureka |
|--------|-----------|--------|
| CAP | CP (保证一致性) | AP (保证可用性) |
| 一致性 | 强一致 | 最终一致 |
| 可用性 | 网络分区时可能不可用 | 网络分区时仍可用 |
| 选主 | ZAB协议 | 无选主 |

### 3. ZooKeeper选主

**ZAB协议流程：**
1. 选举阶段：投票选出Leader
2. 发现阶段：同步数据
3. 同步阶段：Follower同步Leader状态
4. 广播阶段：处理客户端请求

## 五、消息队列

### 1. Kafka

#### 1.1 保证消息顺序

- 同一分区内有序
- 生产者指定分区键
- 消费者单线程消费单分区

#### 1.2 保证消息不丢失

| 阶段 | 措施 |
|------|------|
| 生产者 | acks=all，等待所有副本确认 |
| Broker | 副本因子>=3，min.insync.replicas>=2 |
| 消费者 | 手动提交offset |

#### 1.3 保证消息不重复

- 消费者幂等处理
- 生产者启用幂等性（enable.idempotence=true）
- 使用事务

#### 1.4 高性能设计

| 技术 | 说明 |
|------|------|
| 批量处理 | 批量发送/拉取消息 |
| 顺序读写 | 磁盘顺序写入性能高 |
| PageCache | 利用操作系统缓存 |
| 零拷贝 | sendfile减少数据拷贝 |

## 八、系统设计

### 1. 秒杀系统设计

```
┌─────────────────────────────────────────────────────┐
│                    秒杀系统架构                      │
├─────────────────────────────────────────────────────┤
```

## 九、问题排查

### 1. 常用命令

| 命令 | 用途 |
|------|------|
| `top` | 查看CPU使用情况 |
| `top -Hp <pid>` | 查看进程内线程CPU |
| `jstack <pid>` | 打印线程堆栈 |
| `jmap -histo <pid>` | 查看对象统计 |
| `jmap -dump <pid>` | 导出堆转储 |
| `jstat -gcutil <pid>` | 查看GC情况 |
| `iostat` | 查看IO情况 |

### 2. CPU飙升排查

```
排查流程：
1. top 找到CPU高的进程
2. top -Hp <pid> 找到高CPU线程
3. printf "%x" <tid> 转换线程ID为16进制
4. jstack <pid> | grep <hex-tid> 定位代码位置

常见原因：
- 死循环
- 频繁GC
- 正则表达式回溯
- 加密运算
```

### 3. 内存飙升排查

```
排查流程：
1. jstat -gcutil <pid> 查看GC情况
2. jmap -histo <pid> 查看对象统计
3. jmap -dump 导出堆转储
4. MAT/JProfiler 分析堆转储

常见原因：
- 内存泄漏
- 大对象
- 缓存未清理
```

## 十一、项目经验

### 1. Sharding-JDBC改造

**选型对比：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| MyCAT | 独立部署，无侵入 | 运维复杂，性能损耗 |
| Sharding-JDBC | 轻量，无中心化 | 需要改造代码 |

**改造要点：**
1. 所有SQL必须包含分片键
2. MyBatis/JDBC改造自增ID
3. 事务内强制路由主库
4. 同一线程多次查询路由同一从库
5. 同一接口多线程查询路由同一从库

**UNID生成规则（雪花算法）：**
```
64位 = 1位符号位 + 41位时间戳 + 12位服务进程 + 10位自增序列

时间戳：41位，可用69年
服务进程：12位，支持4096个节点
自增序列：10位，每毫秒生成1024个ID
```

### 2. Push Server升级

**问题：**
- Hazelcast无中心化频繁同步，CPU飙高

**解决方案：**
- Redis替换Hazelcast缓存功能
- Kafka点播替换Hazelcast广播功能
